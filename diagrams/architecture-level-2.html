1邮局有两种：sendMessage 和 Port

三个世界之间的通信不是"打电话"，而是"寄包裹"。但寄包裹有两种方式：一次性快递和长期专线。一个项目里这两种几乎总是同时使用——本项目也不例外。

① 一次性消息 · sendMessage

chrome.runtime.sendMessage / onMessage

一发一收，自动闭合。像寄一个 EMS：寄出去、对方签收、回执回来，整个事就完了。

✓ 适合
  • "切换会话模式"这种瞬间动作
  • "获取历史记录" / "保存设置"等请求
  • 响应一定能在几百毫秒内拿到
✗ 不适合
  • 需要持续推送数据（流式输出）
  • 响应可能要等几十秒（用户可能中途取消）

② 长连接 · Port

chrome.runtime.connect / onConnect

建立一条专线，双方可以多次互发消息，直到任何一方挂断。像装了一台对讲机：开启之后只要不关，随时能继续说。

✓ 适合
  • LLM 流式输出（一条消息陆续吐 token）
  • 用户中途要"停止生成"
  • 需要服务端主动推送多条消息
✗ 不适合
  • 简单的一问一答（杀鸡用牛刀）

本项目的设计原则：简单查询用 sendMessage，流式生成用 Port。这是一条贯穿整个 background.js 的分工线。

两条监听器在 background.js 里长什么样

// 一次性消息：处理 'getHistory' / 'setChatMode' 等
chrome.runtime.onMessage.addListener((request, sender, sendResponse) => {
    handleRuntimeMessage(request, sender)
        .then(sendResponse)
        .catch(...);
    return true; // 关键：让回调能异步
});

// 长连接：只接受名为 'answerStream' 的连接
chrome.runtime.onConnect.addListener((port) => {
    if (port.name !== 'answerStream') return;
    port.onMessage.addListener((request) => {
        handlePortMessage(port, request)...
    });
});

两条监听器各管一摊，互不干扰。如果第三方代码以错误的 port.name 来连，会被直接忽略——这是一个隐式的安全闸门。

2消息格式：一份没写在文档里的"协议"

扩展间消息没有强制格式，由项目自己约定。本项目用的是一种"动作 + 载荷"结构，简单但有效。

一次性消息约定

{
  "action": "getHistory",
  "tabId": 42
}

// 回包：
{
  "status": "ok",
  "history": [...]
}

所有一次性消息都带一个 action 字段。background 里有个大 switch（在 handleRuntimeMessage 里）按 action 分发到具体函数。

Port 消息约定

// content → background：
{
  "action": "startGeneration",
  "tabId": 42,
  "question": "..."
}

// background → content（多次）：
{ "type": "chunk", "text": "你好" }
{ "type": "chunk", "text": "世界" }
{ "type": "done" }

Port 上来回的消息用 type 字段区分（chunk / done / error），因为这里不是"一问一答"，而是事件流。

这套约定没有 TypeScript 类型保护，全靠开发者自律。项目代码里 request.action === 'xxx' 这种判断会出现几十次，加新功能的时候就是新增一个 action 加一个 case。

4流式接力：从 SSE 到 DOM 的四级火箭

"AI 边想边吐字"是怎么做到的？这个体验背后有四个独立段落串成的接力赛——每一段都跑在不同的进程里、用不同的方式传输。

1

OpenAI / Claude → background

请求带 stream: true，服务器以 SSE 协议返回。每一个 token 是一行 data: {...}\n\n。background 用 response.body.getReader() 读 chunk，TextDecoder 拼成完整行，逐行解析。

SSE over HTTP
2

background 内部 · 抽出文本片段

不同 provider 的 SSE 字段各不相同（OpenAI 是 choices[0].delta.content，Anthropic 是 delta.text，Gemini 是 candidates[0].content.parts[0].text）。processStreamLine(apiType, line) 把这堆差异一次性收拢，统一吐出纯文本。

processStreamLine()
3

background → content · Port 推送

每抽出一段文本，立刻 port.postMessage({ type: 'chunk', text })。这是一条已经建好的长连接，几乎零延迟。出错或结束时再发一条 done / error。

port.postMessage()
4

content → DOM · 渲染到聊天气泡

content.js 监听 port.onMessage，每收一条 chunk 就把 text 累加到当前消息字符串里，重新走一次 Markdown 渲染，更新对应气泡的 innerHTML。打字机效果就是这样自然产生的。

marked.parse() + DOM

为什么必须是接力，不能直传？

有人会问：能不能让 content script 直接 fetch OpenAI？

答案是不能。两个原因：

  • 跨域限制：content script 跑在网页所在域下，受 CORS 制约。第三方页面不会允许你给 api.openai.com 发请求。
  • 密钥泄露风险：API Key 一旦进入网页世界，理论上页面里的恶意脚本能嗅探它。所以密钥必须留在 background 这个隔离环境里。

所以你看到了一种典型的"分布式系统"小型化版本：每段只能在最适合它的环境跑，跨段必须接力。

5LLM 适配层：把 5 套不同 API 包成一套

本项目支持 OpenAI Chat / OpenAI Responses / Anthropic / Gemini / Ollama 五种接口。它们的 URL、请求体、流格式、鉴权方式没有一处相同。background 用了一套小型适配层把它们抹平。

Provider URL 形态 鉴权方式 请求体的 messages 字段 流式增量字段
OpenAI Chat /v1/chat/completions Authorization: Bearer [{role, content}] choices[0].delta.content
OpenAI Responses /v1/responses Authorization: Bearer input: [...] + instructions delta.text (event 流)
Anthropic /v1/messages x-api-key + anthropic-version system 单独 / messages delta.text (多事件)
Gemini ?key=xxx (URL 参数) URL 里的 key= contents: [{role, parts}] candidates[0].content...
Ollama /api/chat 无（本机） [{role, content}] message.content (NDJSON)

抹平靠的是几个分发函数

项目里有一组对称的"按 apiType 分支"的小函数。它们形成一条很清晰的处理链：

  • buildEndpointUrl(apiType, ...) — 拼出最终请求 URL（Gemini 的 key 要塞进 query）
  • buildRequestBody(apiType, ...) — 决定调用 buildAnthropicBody / buildGeminiBody / buildResponsesBody 哪一个
  • buildRequestHeaders(apiType, ...) — Bearer 还是 x-api-key
  • processStreamLine(apiType, ...) — 解析流式增量

调用 LLM 的主函数 handleAnswerGeneration 完全不知道下面是哪家——它只调这几个函数。这就是适配器模式（Adapter Pattern）的最朴素实现，没有 class、没有 interface，纯粹靠"按字符串分支的函数族"。

6跨域抓取的旅行：B 站字幕怎么落到 prompt 里

你在 B 站看视频，扩展能拿到字幕——这件事比看起来复杂得多。它揭示了"为什么 background 是扩展的网络出口"。

1

content script 检测到这是 B 站视频页

从 URL 里解析出 BV 号，从 DOM 里抓到 cid（分集 ID）。这一步必须在 content script 里——只有它能读 DOM。

2

content 想去问 B 站的字幕接口

它会想 fetch api.bilibili.com/x/player/wbi/v2?bvid=...。但这个请求跨域——content script 在 www.bilibili.com 上，调 api.bilibili.com 子域时浏览器会要求 CORS 头，B 站不一定给。

3

content 通过消息把请求转给 background

看 shared/videoContext.js 里的 fetchTextViaBackground(url, 'include')。它实际上发了一条 { action: 'fetchText', url, credentials } 的一次性消息给 background。

4

background 替它去发请求

background 在 manifest 声明了 host_permissions: ["<all_urls>"]，不受 CORS 限制（扩展的特权）。它带着用户的 cookie（credentials: 'include'）拿到字幕 JSON，原样回包给 content。

5

content 解析字幕，拼进 prompt

JSON 解析后拿到时间戳 + 文本数组，按顺序拼成 Markdown 形式的对话上下文，作为下一轮提问的 page content。

结论：扩展里所有跨域请求都得 background 出面。这不仅是字幕，PDF 远程下载、图片抓取、模型 API 调用，全都遵守这条规则。

为什么不让 content 自己 fetch？

这背后有一个微妙但关键的细节：扩展在 manifest 里声明的 host_permissions 给的是 background 的特权，不是 content script 的。

Content script 的代码虽然由扩展注入，但它的"网络身份"仍然是宿主网页的——浏览器不会因为它来自扩展就放行 CORS。所以哪怕你在 manifest 里写了 <all_urls>，content script 直接 fetch 跨域接口大概率还是失败。

这是一个新手非常容易踩的坑，也是项目里 fetchTextViaBackground 这种代理函数存在的根本原因。

7网页正文抽取：把 DOM 翻译成 Markdown

用户在新闻页面问"这篇文章在讲什么"，扩展要先把整页正文塞进 prompt。但原始 DOM 太脏了——导航栏、广告、评论区、隐藏元素全在里面。这一步是 content.js 的硬功夫。

挑战在哪

  • 页面结构千奇百怪：有的有 <article>，有的全是 <div class="post">
  • 同样一段文字，不同标签语义不同：<pre> 要保留缩进、<li> 要变成 Markdown 列表
  • 不能简单 document.body.innerText——会把导航、页脚、广告全卷进来
  • 有些元素 display: none，但 innerText 不会管

本项目的策略

  1. 找一个"主内容容器"：先试 <article> / <main>，找不到就退化到 document.body
  2. 深度遍历，按标签类型分类处理（BLOCK_TAGS 集合定义了块级元素）
  3. <pre> / <code> 完整保留缩进，包 ``` 围栏
  4. <h1-6> 转成 Markdown # 标题，<li> 转成 - 列表项
  5. 跳过隐藏节点和明显的"非正文"标签（script / style / nav / footer）

这种做法没有用 Mozilla Readability 那种重型库，而是手写一个简化版。优点是体积小、可控；缺点是个别奇葩页面会抽到杂质。这是个工程权衡问题，不是技术对错问题。

8PDF 抽取：浏览器内置阅读器的特殊路径

PDF 是个特别的存在。Chrome 内置了 PDF 阅读器，但它对扩展几乎是黑盒——你 querySelector 啥也找不到。

Chrome 打开 PDF 时发生了什么

浏览器其实是渲染了一个特殊的 viewer 页面，里面套了个 <embed type="application/pdf">。embed 内部的 PDF 内容由 Chrome 用单独的进程渲染——content script 注入到外层页面，但看不到 embed 内部的文本。所以用普通的 DOM 抽取根本拿不到字。

不行的做法 ✗

document.body.innerText 只能拿到"PDF 文件" 这几个字，因为字本身在 embed 里。

⇒
本项目的做法 ✓

检测到是 PDF 页 → 让 background 重新 fetch(location.href) 拿原始字节 → 用打包进扩展的 pdf.js 在 worker 里解析每一页文本 → 拼成 Markdown 回到 content script。

这条路径上的几个细节

  • 识别 PDF 页：三层判断——document.contentType === 'application/pdf'、URL 以 .pdf 结尾、或者 DOM 里有 <embed type="application/pdf">。任何一条命中就走 PDF 路径。
  • 为什么由 background 取字节：和 B 站字幕同理，content script 跨域受限。background 取到二进制后转成 base64 回包（消息通道里只能传可序列化的值，不能传 ArrayBuffer）。
  • pdf.js 用 worker：解析大 PDF 是 CPU 密集的，必须放 Web Worker 里跑，不然会卡住整个扩展面板。worker 文件路径用 chrome.runtime.getURL('lib/pdf.worker.min.js') 拿到——这是 web_accessible_resources 必须声明的原因。
  • 缓存：同一个 URL 半小时内不重复解析。PDF 解析比较贵，缓存收益明显。

9Slash 指令：一行 / 触发的小宇宙

你在输入框打 /，弹出一个候选列表，选中后自动填入预设的提示词。这是个看着小、设计上有意思的小功能。

/summ
/summarize 五条要点总结
/summary （用户自定义）

这套机制的三个有意思的点

① 共享定义

指令模板放在 shared/slashCommands.js，三个世界共用一份。新增内置指令只要改这一个文件。

② 用户可改

用户在 options 页能增删改自定义指令，存到 chrome.storage.sync.slashCommands。normalizeSlashCommands 做了校验：去重、过滤非法字符、忽略空 prompt。

③ 平滑升级

mergeNewDefaultSlashCommands 解决一个细节：版本升级新增内置指令时，老用户原本保存的列表里没有它——这个函数把白名单里的内置指令补回去，又不覆盖用户自己改过的。

触发流程

1

用户在输入框打字

content.js 监听输入框 input 事件。每次值变化都正则匹配 /^\/([\w-]*)$/。

2

匹配上就拉出候选列表

对当前所有指令做 prefix 过滤，按命中度排序，渲染成弹层。autoSubmit 标记的指令选中后会立刻发送，不需要用户再按回车。

3

选中后替换输入框内容

把 /xxx 替换成对应的 prompt 全文，光标移到末尾。后续走的就是普通的"问答"流程，跟你直接打字没区别。

Slash 指令本质上是客户端层面的 prompt 模板系统。它不需要后端、不依赖模型、不增加上下文成本——纯前端字符串替换，但显著降低了用户用扩展的认知负担。
