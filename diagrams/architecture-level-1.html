PageLens AI 是怎么搭起来的

把这个项目从"一坨能跑的代码"还原成"一张能讲清楚的图"。本页不讲实现细节，只讲结构——每一块是什么、彼此怎么连、为什么要这么分。

共约 11,000 行代码 · 6 个会话模式 · 5 种 LLM 接口 · 3 个执行环境 第 1 层 / 共 N 层

一它到底是个什么东西

一句话定义：一个浏览器扩展，把"当前网页 / PDF / 视频字幕"变成 AI 的上下文，让你可以基于眼前的内容直接提问。

用工程语言再说一次：

  • 它是一个 Chrome Manifest V3 扩展（不是 Web App、不是 App、不是 CLI 工具）。
  • 它不训练模型、不跑模型，只是一个聪明的"中间人"——把网页内容拿出来，拼成 prompt，转发给你配置的 OpenAI / Claude / Gemini / Ollama。
  • 它附带一个本地 Python 小服务，专门负责"把对话记录写到磁盘"这件浏览器自己干不了的事。

所以理解这个项目，本质上就是理解三件事：① 浏览器扩展的三个世界 ② 一次问答的数据流 ③ 浏览器为什么需要一个本地搭子。

二三个世界：扩展的骨架

这是 MV3 扩展的核心心智模型。整个项目所有文件，都属于下面三个世界之一。它们运行在不同的 JS 进程里，不能直接互相调用函数，只能发消息。

BACKGROUND

后台世界

background.js · ~3000 行
  • 调用 LLM 的 HTTP 请求
  • 管理所有标签页的会话状态
  • 路由消息给前台 / 面板
  • 跨域抓取（B站字幕等）
✓ 能跨域请求任何 URL ✗ 看不到网页 DOM ✗ 随时会被浏览器杀掉
CONTENT

页面世界

content.js · ~3500 行
  • 渲染右侧聊天面板 DOM
  • 抽取网页正文 / PDF
  • 检测视频页、注入悬浮球
  • 处理 Slash 指令、概念标注
✓ 能读写当前网页 DOM ✗ 不能跨域请求 ✗ 看不到页面里的 JS 变量（隔离世界）
UI PAGES

独立页面世界

popup/ · options/
  • 点击图标弹出的小窗口
  • 设置页（API 密钥、模型）
  • 独立的 HTML 页面
✓ 普通网页能干的都能干 ✗ 关掉就没了，不持久

为什么要分成三个？

这不是"代码组织"问题，而是浏览器安全模型的强制要求。如果让网页里随便一段 JS 就能调你扩展的功能、读你扩展的存储、用你扩展的 API 密钥去请求任何地址——那扩展就成了攻击者的免费跳板。

所以浏览器把它们隔离开：每个世界有自己的能力边界，要协作必须通过消息通道明确传递。这是这类项目最难的部分，也是 AI 写代码最容易出错的部分。

三一次问答的完整旅程

把整个项目当成一个剧本来看。从你按下快捷键，到屏幕上吐出第一个字，中间发生了什么？

1

用户按 Cmd+Shift+K

浏览器收到快捷键，根据 manifest 里的注册，把"toggle-panel"事件丢给后台。

Browser → Background
2

后台通知页面世界："给用户开面板"

背景世界自己看不到 DOM，所以它发一条消息给当前 tab 的 content script。

Background → Content
3

页面世界渲染聊天面板

content.js 在页面右侧插入一个 DOM 容器，里面装着输入框、消息列表、模式切换按钮。这是用户能看到的全部 UI。

Content (DOM)
4

用户输入问题，按回车

页面世界根据当前"会话模式"决定要不要带上下文：网页模式抽正文，视频模式抓字幕，纯聊模式啥也不带。

Content (准备 payload)
5

把"问题 + 上下文"递交给后台

因为只有后台能跨域请求 OpenAI / Claude 等服务，所以页面世界必须通过消息把活儿转交出去。这里用的是长连接（Port），方便后续流式接收。

Content → Background
6

后台拼 prompt、调 LLM

根据用户在设置页配置的 API 类型（OpenAI / Anthropic / Gemini / Ollama / Responses），构造对应格式的请求体，开启 stream: true，开始接收 SSE 流。

Background → 外部 LLM
7

边收边推回前端，逐字显示

每收到一个 token chunk，后台就通过那条长连接推回页面世界，content.js 把它追加到消息气泡里。这就是"打字机效果"的来源。

Background → Content (流)
8

对话结束，按需归档

如果当前是"入库"模式，后台把整轮对话拼成 Markdown，POST 给本地 127.0.0.1:8765 的 Python 桥接服务，由它写到磁盘。如果是"临时"模式，就只留在内存里。

Background → 本地 Python 服务

注意一个反直觉的事实：看起来是"用户在跟 AI 聊"，实际上消息要在三个世界之间至少跳 6 次。这就是为什么调试扩展比调试网页麻烦得多——bug 可能藏在任何一段消息通道里。

四六种会话模式：两个开关的笛卡尔积

项目宣称有 6 种模式，听起来很复杂。其实只是两个独立开关相乘的结果——理解这一点，整个产品逻辑就通了。

开关 A：上下文从哪来？

  • 网页 — 抽取整页正文（或 PDF 文本）
  • 视频 — 抓 B 站 / YouTube 字幕
  • 纯聊 — 不带任何页面内容

开关 B：要不要存盘？

  • 入库 — 写入本地 Markdown 日志
  • 临时 — 只活在当前会话里
模式上下文持久化典型场景
网页 + 入库整页正文写日志读一篇要复习的长文章
网页 + 临时整页正文不写随手问一句就关掉
视频 + 入库视频字幕写日志看长课程做笔记
视频 + 临时视频字幕不写看个短视频问个梗
纯聊 + 入库无写日志当一个普通 AI 助手用
纯聊 + 临时无不写问个无关的小问题

这套 6 模式定义集中在 shared/chatModes.js 里，同一份配置被三个世界共用——这就是 shared/ 文件夹存在的意义：避免三个世界各写一份导致漂移。

五模块地图：把 11000 行分成 8 块

不要把整个项目当成"两个大文件 + 一堆杂项"。它实际上是 8 个职责相对独立的子系统拼起来的。

① 扩展骨架

manifest.json

声明扩展的能力清单：要哪些权限、注入哪些脚本、注册哪个快捷键。整个项目的"配置中枢"。

② 后台调度中心

background.js

会话状态管理、消息路由、LLM 请求构造、流式响应转发。它是整个扩展的"大脑"。

③ 页面 UI + 内容抽取

content.js · styles/content.css

面板 DOM、Markdown 渲染、用户输入处理、网页正文抽取、悬浮球、概念标注。所有"用户能看到摸到"的东西。

④ 设置页与弹窗

options/ · popup/

独立的 HTML 页面。设置页负责 API 配置和连通性测试，弹窗负责快速切换模式。

⑤ 共享定义层（关键）

shared/chatModes.js · mentorModes.js · slashCommands.js · skills.js

三个世界都需要的常量与枚举。这一层的存在是为了消除重复。AI 写代码时最容易在这里复制粘贴出三份不同步的版本。

⑥ 内容抽取专项

shared/pdfParser.js · videoContext.js · imageInput.js

三种"特殊上下文"的抓取：PDF 用 pdf.js 解析文本，视频要懂 B 站 / YouTube 的字幕接口，图片要做 base64 编码。每一个都是独立的小工程。

⑦ 第三方库

lib/marked.min.js · pdf.min.js · pdf.worker.min.js

Markdown 渲染和 PDF 解析。直接打包进扩展，不依赖 CDN——因为有些用户访问的页面可能屏蔽外部资源。

⑧ 本地桥接服务

tools/session_log_bridge.py

一个独立的 Python 程序，跑在用户自己电脑上，监听 127.0.0.1:8765。它和扩展之间通过 HTTP 通信，不属于扩展本身。

六边界故事：浏览器为什么需要一个本地搭子

这是整个项目最值得玩味的设计决策。它揭示了浏览器的一个根本限制——也是你理解 Web 安全模型的好机会。

浏览器扩展能做的

读 DOM、调远程 API、用 chrome.storage 存几 MB 数据、显示通知。

╳ 不能跨越

浏览器扩展不能做的

把数据写到任意本地路径（比如 ~/Documents/笔记/）、读你硬盘上的其他文件、调系统命令。

为什么不行？浏览器从设计上就把 Web 内容当成"潜在敌对方"对待——任何网页或扩展都不能随便动你的文件系统，否则一个恶意扩展就能加密你所有文档。

那怎么办？项目作者的解法很经典：在用户自己的电脑上跑一个小型 HTTP 服务，扩展通过 fetch('http://127.0.0.1:8765/log-session') 把数据发给它，由这个 Python 程序代为写盘。

这就把"危险操作"的责任明确转移到了一个用户主动启动、能随时关掉的程序里。这个模式在很多桌面级 Web 工具里都用过——比如 Obsidian 的本地 REST 插件、Logseq 的同步桥、各种 IDE 的浏览器扩展。
