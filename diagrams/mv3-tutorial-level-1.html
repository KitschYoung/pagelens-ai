什么是 MV3 浏览器扩展

从"它和网站、App 有什么区别"开始，把 Chrome MV3 扩展的心智模型搭起来。这一层不写一行代码，只解释"它是什么、为什么这么设计"。

预计阅读 12–15 分钟 · 互动元素 · SVG 解剖图 · 自测 第 1 层 / 共 N 层

1先问一个钝钝的问题：扩展到底是个啥

在解释 MV3 之前，得先把"浏览器扩展"这个东西本身搞清楚。它不是网站，也不是手机 App，但又像两者的混合。下面这个类比能省你很多脑力。

🔌

浏览器扩展 ≈ 给浏览器装的一个外接零件

网站是客人——你随便打开一个，看完就走，浏览器对它有严格防备。
App 是独立公民——它有自己的进程、自己的窗口，能动你硬盘里的文件。
扩展是外聘员工——浏览器签了合同雇它，给它发工牌（权限），它能帮你处理网页，但只能在浏览器画的圈里活动。

能做什么普通网站浏览器扩展桌面 App
读你正在看的网页内容不能能（被授权时）不能
同时操作多个标签页不能能不能
跨域请求任意 URL受 CORS 限制能（声明权限后）能
注册全局快捷键不能能能
读写本地任意文件不能不能能
关闭浏览器后还能跑不能不能能

所以扩展的能力恰好夹在网站和 App 之间：比网站强，但还在浏览器的管辖之内。它不能逃出浏览器、不能动你硬盘——这是它能被信任地装进千万人电脑里的前提。

下面这张图把一个 Chrome 浏览器从外向内拆成五层。扩展同时住在好几层里——这是它最反直觉的地方，也是 MV3 心智模型的起点。

隔离世界里 网页里的变量它看不见，反之亦然 SERVICE WORKER 进程 background.js 扩展的"大脑" 能跨域请求 看不到网页 DOM EXTENSION PAGES popup / options / 独立页面 点扩展图标弹出的小窗口 设置页（chrome-extension://...） 关掉就没了 消息通信
Service Worker扩展的后台大脑，单独进程
Content Script注入到网页里的"耳目"
Extension Pages扩展自己的 HTML 页面
工具栏图标用户唯一直接接触的入口

一个扩展不是"一个进程"，而是同时存在于多个进程的一组协作者。理解这一点，是 MV3 入门最大的台阶。

3三个世界：能力与限制

把上面那张图压缩一下：扩展永远活在三类地方。每类地方都有自己的"超能力"和"残疾"——它们恰好互补，所以才必须协作。

BACKGROUND

后台 · Service Worker

独立进程
  • 调远程 API（任何域名）
  • 响应快捷键、扩展图标点击
  • 在标签页之间共享状态
  • 定时任务（alarms）
✓ 能跨域 fetch 任何 URL
✗ 看不到任何网页的 DOM
✗ 几十秒不活跃就被杀
CONTENT

页面 · Content Script

注入到网页进程
  • 读写当前页面的 DOM
  • 响应页面事件（鼠标、滚动）
  • 给页面加 UI（如悬浮球）
✓ 能改当前网页的内容
✗ 不能跨域请求
✗ 看不到页面里的 JS 变量
UI PAGES

扩展页面 · Popup / Options

独立 HTML 页面
  • 普通的网页，但跑在扩展环境里
  • 点击图标弹出的小窗
  • 设置页、调试页等
✓ 普通网页能干的都行
✗ 关掉窗口就没了
✗ 不能直接读其他网页的 DOM

为什么要分得这么细？

因为浏览器要解决一个矛盾：扩展需要能力，但不能滥用能力。

如果让 content script 也能跨域请求，那任何网页里的恶意代码都可能利用扩展的密钥发请求。如果让 service worker 直接动 DOM，那它就能在你完全不察觉的情况下改写银行网站的内容。所以浏览器把"可以读网页"和"可以联网"故意拆到不同进程，要协作就必须显式发消息——这一拆，攻击面就小了一大圈。

5一个扩展长什么样：文件结构

所有 MV3 扩展，从最简单到最复杂，都遵循同一种基本骨架。打开一个扩展的源码，你大概率会看到下面这样的目录。

my-extension/ ├── manifest.json # ★ 必须有，扩展的"身份证" ├── background.js # Service Worker（大脑） ├── content.js # Content Script（注入网页的耳目） ├── popup/ │ ├── popup.html # 点图标弹出的小窗 │ ├── popup.js │ └── popup.css ├── options/ │ ├── options.html # 设置页（独立 tab 打开） │ └── options.js ├── icons/ │ └── icon128.png # 工具栏 / 商店里显示的图标 └── _locales/ # 多语言（可选） └── zh_CN/messages.json

红色的 manifest.json 是唯一必须有的文件——它就像扩展的身份证 + 雇佣合同，告诉浏览器：

  • 我叫什么、什么版本
  • 我要哪些权限（要不要读所有网页？要不要操作 cookie？）
  • 我的后台脚本是哪一个
  • 我要往哪些网址注入 content script
  • 我的图标和设置页在哪

其它所有 JS / HTML / CSS / 图片，本质上都是 manifest.json 里被引用到的"零件"。没有 manifest 的一堆文件，浏览器不认。

6权限模型：你被授权了什么

扩展要做事，得先在 manifest 里"申报"自己要什么权限。安装时浏览器会把这些权限以人话告诉用户。下面是常见权限按"危险等级"分的三档。

LOW · 低风险

storage

能用 chrome.storage 存自己的几 MB 数据。安装时浏览器不会特别警告。

LOW · 低风险

alarms

能注册定时任务，类似 setTimeout 但 service worker 死了也能唤醒。

LOW · 低风险

notifications

能弹系统通知。

MID · 需提示

activeTab

用户主动点击扩展图标时，授权它读当前这一个标签页。比下面那个 tabs 安全得多。

MID · 需提示

tabs

能查询所有打开的标签页（标题、URL）。能拿到你浏览历史的快照，所以浏览器会警告。

MID · 需提示

scripting

动态地往任意页面注入脚本。MV3 里很多扩展都需要它。

HIGH · 强警告

<all_urls>

声明 host_permissions 为所有网站，意味着你能读取和修改用户访问的任何网页。安装时会弹巨大的警告框。

HIGH · 强警告

cookies

能读写用户的所有 cookies。这等于能拿登录态，所以是高敏。

HIGH · 强警告

webRequest

能拦截、修改网络请求。功能极强，也是恶意扩展最爱的权限。

为什么用户应该看权限

很多人安装扩展时随手点"添加"。但权限就是合同——你装一个声明了 <all_urls> 和 cookies 的扩展，等于给它你浏览器里所有登录态的副本。所以一个良好的扩展会尽量声明最小权限集，比如优先用 activeTab 而不是 tabs。

7生命周期：service worker 不是常驻进程

这是 MV3 最反直觉、也最容易让初学者踩坑的事实——你的后台脚本随时会被浏览器杀掉。它不是 MV2 那种长期开着的页面。

用户安装
空闲 30s
收到消息
空闲 30s
快捷键触发
启动初始化完成，监听消息
休眠没人理它，被浏览器杀掉，内存里的变量全没
复活有事件来了，重新启动
再次休眠之前的状态全丢，但 chrome.storage 里的还在
再启动
运行中 已被杀死

这意味着什么

🚫 不能做的

  • 在 background.js 里用全局变量保存重要状态
  • 用 setInterval 跑常驻轮询
  • 假设"上次写过的变量这次还在"

✅ 应该做的

  • 状态写进 chrome.storage
  • 定时任务用 chrome.alarms
  • 每次脚本启动都重新订阅事件监听器

Google 这么设计的理由是：省内存。MV2 时代每个扩展都常驻一个后台页面，开 50 个扩展就是 50 个隐藏 tab 在吃内存。MV3 改成事件驱动后，扩展只在真正有事的时候才占资源。代价就是开发者必须接受"我的代码随时会从头跑一遍"。

8MV2 vs MV3：到底改了什么

你会在网上看到大量"为什么 MV3 这么糟"的抱怨。这一节快速给你交底——只看两个版本最关键的差异，不站队。

MV2 · 旧时代

2012 – 2024
  • 后台是一个常驻 HTML 页面，永远活着
  • 能用 webRequest 阻塞修改网络请求（强力广告拦截器靠这个）
  • 能远程加载并执行任意 JS 代码
  • 权限审核相对宽松
→

MV3 · 现行

2021 – 至今
  • 后台改成 Service Worker，事件驱动、随时被杀
  • webRequest 阻塞模式废除，广告拦截改用 declarativeNetRequest（只能按预定规则匹配）
  • 所有代码必须打包进扩展，不能远程加载
  • 更严的权限审核和 CSP

争议的核心是什么

Google 说：MV3 让扩展更省内存、更安全、更难作恶。
批评者说：MV3 顺便削弱了 uBlock Origin 这类强力广告拦截器，而 Google 自己卖广告，所以这"碰巧"利好 Google 的广告业务。

对你这一层来说不重要。你只需要知道：MV3 是 Chrome 当前唯一支持的扩展版本，所以新写扩展只能用 MV3。本项目（PageLens AI）的 manifest 里写着 "manifest_version": 3，这就是它身处的时代。

Q1一个 MV3 扩展必须包含哪个文件？
background.js
content.js
manifest.json
popup.html
只有 manifest.json 是必须的。其它脚本都是 manifest 引用到的"零件"，没引用就不存在。
Q2用户访问 example.com，扩展要读这个页面的 H1 标题。这件事应该谁来做？
Service Worker（background.js）
Content Script（content.js）
Popup 页面
Options 页面
只有 content script 能直接读页面 DOM。其它三个都没有访问网页 DOM 的能力。
Q3扩展要调用 OpenAI 的 API（api.openai.com），这件事应该谁来做？
Service Worker（background.js）
Content Script（content.js）
谁都行，没区别
网页本身
跨域请求必须放在后台。Content script 受 CORS 限制，跨域请求大概率被拒。
Q4下面哪种做法在 MV3 里会出问题？
把用户配置存进 chrome.storage
在 service worker 里监听快捷键
在 service worker 顶层用一个全局变量保存登录状态
在 content script 里给页面加一个悬浮球
Service worker 随时会被杀掉，全局变量不可靠。状态必须落到 chrome.storage。
Q5"扩展要读取并更改你访问的所有网站上的数据"——这条警告对应 manifest 里的什么声明？
"permissions": ["storage"]
"action": { ... }
"background": { "service_worker": "..." }
"host_permissions": ["<all_urls>"]
这条警告专门由 host_permissions 触发，表示扩展能在哪些域名上注入脚本和发请求。
